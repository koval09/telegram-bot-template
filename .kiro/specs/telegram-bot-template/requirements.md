# Требования: Шаблонный Telegram-бот (telegram-bot-template)

## Введение

Универсальный шаблонный Telegram-бот, на базе которого можно строить боты разного назначения. Шаблон предоставляет готовую инфраструктуру: регистрацию пользователей, хранение данных в базе, модуль профиля, интеграцию с TON-кошельком через TON Connect, защищённую админ-панель с модерацией (бан/кик/мут), а также набор дополнительных модулей (реферальная система, рассылки, статистика, мультиязычность, платежи, антиспам, проверка подписок на каналы). Модули подключаются поэтапно и по необходимости — каждый новый проект может включать только нужные ему компоненты без переписывания ядра.

Документ описывает ЧТО должен уметь шаблон (поведение, интерфейсы, гарантии) и НЕ описывает КАК это реализовано — конкретный выбор библиотек, ORM, фреймворка и схем БД выносится в документ проектирования (design).

## Глоссарий

- **Telegram_Бот**: основное приложение, взаимодействующее с Telegram Bot API и обслуживающее пользователей.
- **База_Данных**: постоянное хранилище данных пользователей, профилей, ролей, логов и состояния модулей.
- **Модуль_Регистрации**: компонент, создающий запись о пользователе при первом взаимодействии с ботом.
- **Модуль_Профиля**: компонент, предоставляющий пользователю просмотр и редактирование собственных данных.
- **TON_Коннектор**: компонент, реализующий подключение внешнего TON-кошелька пользователя по протоколу TON Connect 2.
- **Админ_Панель**: набор команд и интерфейсов, доступных только авторизованным администраторам.
- **Менеджер_Пользователей**: компонент, выполняющий операции модерации (бан, кик, мут, разбан, размут) над пользователями.
- **Модуль_Авторизации**: компонент, проверяющий, является ли текущий пользователь администратором и имеет ли он право на действие.
- **Журнал_Действий**: компонент, сохраняющий все действия администраторов и важные события системы.
- **Реферальная_Система**: компонент, генерирующий реферальные ссылки и учитывающий приглашения.
- **Модуль_Рассылки**: компонент, отправляющий сообщения всем или выбранной группе пользователей.
- **Модуль_Статистики**: компонент, собирающий и отдающий агрегированные метрики по пользователям и активности.
- **Модуль_Локализации**: компонент, возвращающий тексты интерфейса на выбранном языке пользователя.
- **Модуль_Антиспам**: компонент, ограничивающий частоту сообщений и проверяющий нового пользователя капчей.
- **Модуль_Подписок**: компонент, проверяющий подписку пользователя на заданные каналы/чаты Telegram.
- **Модуль_Платежей**: компонент, обрабатывающий оплату через Telegram Stars и/или TON.
- **Менеджер_Состояний**: компонент, хранящий пошаговые диалоги (FSM) пользователя.
- **Кэш**: быстрое хранилище (Redis или аналог) для FSM, лимитов антиспама и временных данных.
- **Конфигурация**: файл/переменные окружения с секретами и настройками (токен бота, DSN базы, список админов, ключи TON).
- **Администратор**: пользователь, идентификатор Telegram которого присутствует в таблице ролей со значением `admin` или `superadmin`.
- **Суперадминистратор**: администратор, имеющий право назначать и снимать других администраторов.
- **Забаненный_Пользователь**: пользователь, у которого в Базе_Данных поле `status` равно `banned`.
- **Заглушённый_Пользователь**: пользователь, у которого в Базе_Данных поле `status` равно `muted` и не истёк срок мута.

## Требования

### Требование 1: Регистрация пользователя и хранение в Базе_Данных

**User Story:** Как владелец бота, я хочу, чтобы каждый новый пользователь автоматически регистрировался в Базе_Данных, чтобы я мог вести учёт аудитории и строить на этом другую логику.

#### Acceptance Criteria

1. WHEN пользователь впервые отправляет команду `/start` (запись с соответствующим `telegram_id` отсутствует в Базе_Данных), THE Модуль_Регистрации SHALL в течение 3 секунд создать в Базе_Данных запись, содержащую поля `telegram_id`, `username`, `first_name`, `last_name`, `language_code`, `created_at` (метка времени UTC), `status=active`.
2. WHEN пользователь, запись которого уже существует в Базе_Данных, отправляет любое сообщение, THE Модуль_Регистрации SHALL в течение 3 секунд обновить поле `last_seen_at` текущей меткой времени UTC.
3. IF при получении сообщения от существующего пользователя значения полей `username`, `first_name`, `last_name` или `language_code`, переданные Telegram, отличаются от сохранённых в Базе_Данных, THEN THE Модуль_Регистрации SHALL обновить в Базе_Данных только те поля, значения которых изменились, сохранив остальные поля без изменений.
4. IF операция записи или обновления в Базе_Данных завершается ошибкой, THEN THE Модуль_Регистрации SHALL выполнить до 3 повторных попыток с интервалом не менее 1 секунды перед признанием операции неуспешной, не сохраняя частично записанные данные.
5. IF после исчерпания повторных попыток операция записи или обновления в Базе_Данных остаётся неуспешной, THEN THE Telegram_Бот SHALL вернуть пользователю сообщение с уведомлением об ошибке регистрации и зафиксировать запись об ошибке в Журнале_Действий с указанием `telegram_id` пользователя и причины ошибки.
6. THE База_Данных SHALL обеспечивать уникальность записи пользователя по полю `telegram_id`, отклоняя вставку записи с уже существующим значением `telegram_id`.
7. WHEN пользователь впервые отправляет команду `/start` с параметром реферальной ссылки, содержащим `telegram_id` существующего в Базе_Данных пользователя и отличный от `telegram_id` отправителя, THE Модуль_Регистрации SHALL сохранить идентификатор пригласившего в поле `referrer_id` создаваемой записи пользователя.
8. IF команда `/start` с параметром реферальной ссылки получена от уже зарегистрированного пользователя, ИЛИ параметр содержит `telegram_id`, равный `telegram_id` отправителя, ИЛИ `telegram_id`, отсутствующий в Базе_Данных, THEN THE Модуль_Регистрации SHALL проигнорировать параметр реферальной ссылки и оставить поле `referrer_id` без изменений.

### Требование 2: Профиль пользователя

**User Story:** Как пользователь бота, я хочу видеть и редактировать свой профиль, чтобы управлять своими данными.

#### Acceptance Criteria

1. WHEN пользователь отправляет команду `/profile`, THE Модуль_Профиля SHALL вернуть сообщение с полями: `telegram_id`, отображаемое имя, язык, дата регистрации, статус, привязанный TON-адрес (если есть), количество приглашённых рефералов (если Реферальная_Система включена).
2. WHEN пользователь нажимает кнопку «Сменить язык» в профиле, THE Модуль_Профиля SHALL показать список поддерживаемых языков и сохранить выбор в поле `language_code`.
3. WHEN пользователь нажимает кнопку «Отвязать кошелёк» в профиле и у него привязан TON-адрес, THE Модуль_Профиля SHALL удалить привязку и вернуть подтверждающее сообщение.
4. IF пользователь с `status=banned` вызывает `/profile`, THEN THE Telegram_Бот SHALL вернуть сообщение о блокировке и не отдавать данные профиля.
5. THE Модуль_Профиля SHALL отображать только данные текущего пользователя и не предоставлять доступ к данным других пользователей.

### Требование 3: Подключение TON-кошелька через TON Connect

**User Story:** Как пользователь, я хочу подключить свой TON-кошелёк к боту через TON Connect, чтобы бот мог верифицировать владение адресом и использовать его в будущих сценариях.

#### Acceptance Criteria

1. WHEN пользователь вызывает команду `/connect_wallet`, THE TON_Коннектор SHALL сгенерировать ссылку/QR-код на сессию TON Connect 2 и вернуть её пользователю.
2. WHEN TON-кошелёк подтверждает подключение через TON Connect, THE TON_Коннектор SHALL верифицировать подпись (`proof`) и сохранить в Базе_Данных поля `ton_address`, `ton_wallet_name`, `ton_connected_at`.
3. IF верификация подписи TON Connect не прошла, THEN THE TON_Коннектор SHALL отклонить подключение, не записывать адрес в Базу_Данных и вернуть пользователю сообщение об ошибке.
4. WHEN пользователь вызывает команду `/disconnect_wallet` и у него привязан кошелёк, THE TON_Коннектор SHALL удалить `ton_address` и связанные поля в Базе_Данных и завершить сессию TON Connect.
5. WHERE сессия подключения TON Connect не завершена пользователем в течение 10 минут, THE TON_Коннектор SHALL пометить сессию как истёкшую и освободить связанные ресурсы.
6. THE TON_Коннектор SHALL поддерживать одновременно только один активный привязанный кошелёк на одного пользователя.

### Требование 4: Авторизация администраторов

**User Story:** Как владелец бота, я хочу, чтобы админ-панель была доступна только доверенным администраторам, чтобы исключить несанкционированный доступ к функциям модерации.

#### Acceptance Criteria

1. THE Модуль_Авторизации SHALL определять пользователя как Администратор только если запись о нём в Базе_Данных имеет роль `admin` или `superadmin`.
2. WHEN любая команда Админ_Панели вызывается пользователем, THE Модуль_Авторизации SHALL проверить его роль до выполнения действия.
3. IF команда Админ_Панели вызывается пользователем, не являющимся Администратором, THEN THE Telegram_Бот SHALL ответить одинаковым сообщением «Команда не найдена» и записать попытку в Журнал_Действий с уровнем `warning`.
4. THE Модуль_Авторизации SHALL считать список начальных Суперадминистраторов доверенным только если он задан в Конфигурации через переменную окружения, а не через команды бота.
5. WHEN Суперадминистратор вызывает команду назначения нового администратора, THE Модуль_Авторизации SHALL назначить роль `admin` указанному пользователю и записать изменение в Журнал_Действий.
6. IF обычный Администратор пытается назначить или снять другого Администратора, THEN THE Модуль_Авторизации SHALL отклонить операцию и записать попытку в Журнал_Действий.
7. WHEN роль пользователя изменяется (назначение/снятие), THE Модуль_Авторизации SHALL инвалидировать кэш его прав в Кэше в течение 5 секунд.

### Требование 5: Управление пользователями (бан, кик, мут)

**User Story:** Как Администратор, я хочу банить, кикать и мутить пользователей, чтобы защитить сообщество бота от нежелательного поведения.

#### Acceptance Criteria

1. WHEN Администратор вызывает команду бана с идентификатором пользователя и причиной, THE Менеджер_Пользователей SHALL установить в Базе_Данных для целевого пользователя `status=banned`, сохранить `banned_at`, `banned_by`, `ban_reason`.
2. WHILE у пользователя `status=banned`, THE Telegram_Бот SHALL игнорировать все его входящие сообщения и команды, кроме возврата однократного уведомления о блокировке не чаще одного раза в 24 часа.
3. WHEN Администратор вызывает команду мута с идентификатором пользователя и длительностью, THE Менеджер_Пользователей SHALL установить `status=muted` и `muted_until=now + длительность`.
4. WHILE у пользователя `status=muted` и текущее время меньше `muted_until`, THE Telegram_Бот SHALL игнорировать сообщения пользователя, отвечая только стандартным уведомлением о муте не чаще одного раза в 10 минут.
5. WHEN текущее время становится больше `muted_until`, THE Менеджер_Пользователей SHALL автоматически вернуть пользователю `status=active` при его следующем обращении.
6. WHEN Администратор вызывает команду кика в групповом чате, где бот имеет права администратора, THE Менеджер_Пользователей SHALL удалить целевого пользователя из чата через Telegram Bot API и записать действие в Журнал_Действий.
7. WHEN Администратор вызывает команду разбана с идентификатором пользователя, THE Менеджер_Пользователей SHALL установить `status=active` и очистить поля `banned_at`, `ban_reason`.
8. IF Администратор пытается забанить/замутить другого Администратора или Суперадминистратора, THEN THE Менеджер_Пользователей SHALL отклонить операцию и вернуть сообщение «Недостаточно прав».
9. IF указанный идентификатор пользователя отсутствует в Базе_Данных, THEN THE Менеджер_Пользователей SHALL вернуть сообщение «Пользователь не найден» и не изменять состояние.

### Требование 6: Журнал действий и аудит

**User Story:** Как Суперадминистратор, я хочу видеть историю всех действий администраторов и важных системных событий, чтобы расследовать инциденты и контролировать команду.

#### Acceptance Criteria

1. WHEN Администратор выполняет действие модерации из фиксированного перечня (бан, мут, кик, разбан, назначение роли), THE Журнал_Действий SHALL в течение 2 секунд сохранить запись, содержащую поля `actor_id` (Telegram ID администратора), `target_id` (Telegram ID цели), `action` (тип действия из перечня), `reason` (строка длиной до 500 символов, допускается пустое значение) и `created_at` (временная метка в UTC в формате ISO 8601).
2. WHEN в системе происходит ошибка подключения к Базе_Данных, TON Connect или Telegram API, THE Журнал_Действий SHALL сохранить запись с уровнем `error`, источником ошибки (один из: База_Данных, TON Connect, Telegram API) и текстом ошибки длиной до 1000 символов (с усечением более длинных сообщений и пометкой об усечении).
3. WHEN Суперадминистратор вызывает команду просмотра Журнал_Действий, THE Журнал_Действий SHALL вернуть записи, отсортированные по `created_at` в порядке убывания, страницами по 50 записей, с указанием номера текущей страницы, общего количества страниц и общего количества записей, а время отклика SHALL не превышать 3 секунды.
4. THE Журнал_Действий SHALL хранить записи не менее 90 дней с момента создания и SHALL автоматически удалять записи старше 90 дней при ежедневной проверке, регистрируя факт удаления отдельной записью с уровнем `info`.
5. IF команду просмотра Журнал_Действий вызывает пользователь, не являющийся Суперадминистратором, THEN THE Журнал_Действий SHALL отклонить запрос, вернуть сообщение об отсутствии прав доступа и не раскрывать содержимое журнала.
6. IF сохранение записи в Журнал_Действий завершается ошибкой Базы_Данных, THEN THE Журнал_Действий SHALL повторить попытку до 3 раз с интервалом 1 секунда между попытками, и при исчерпании всех попыток SHALL направить уведомление Суперадминистратору, указав тип несохранённого события и время возникновения, без блокировки исходного действия модерации.

### Требование 7: Реферальная система

**User Story:** Как владелец бота, я хочу иметь встроенную реферальную систему, чтобы стимулировать пользователей приглашать новых.

#### Acceptance Criteria

1. WHERE Реферальная_Система включена в Конфигурации, THE Telegram_Бот SHALL предоставлять пользователю команду `/ref`, возвращающую персональную ссылку формата `https://t.me/<bot_username>?start=ref_<telegram_id>`.
2. WHEN новый пользователь регистрируется по реферальной ссылке, THE Реферальная_Система SHALL увеличить счётчик `referrals_count` у пригласившего на 1.
3. IF пользователь переходит по собственной реферальной ссылке, THEN THE Реферальная_Система SHALL не засчитывать приглашение и не менять счётчики.
4. WHEN пригласивший пользователь запрашивает список рефералов, THE Реферальная_Система SHALL вернуть общее количество приглашённых и дату последнего приглашения.

### Требование 8: Рассылка сообщений

**User Story:** Как Администратор, я хочу отправлять сообщения всем или выбранной группе пользователей, чтобы делать объявления.

#### Acceptance Criteria

1. WHEN Администратор запускает рассылку с текстом и фильтром (все / активные / по языку), THE Модуль_Рассылки SHALL поставить задание в очередь и начать отправку.
2. THE Модуль_Рассылки SHALL отправлять сообщения с соблюдением лимита Telegram Bot API не более 30 сообщений в секунду.
3. IF Telegram Bot API возвращает ошибку `403 Forbidden: bot was blocked by the user` при отправке, THEN THE Модуль_Рассылки SHALL установить у получателя флаг `is_blocked_bot=true` и продолжить рассылку остальным.
4. WHEN рассылка завершается, THE Модуль_Рассылки SHALL вернуть инициатору отчёт: всего получателей, доставлено, не доставлено, заблокировали бота.
5. IF Администратор запускает новую рассылку, пока предыдущая не завершена, THEN THE Модуль_Рассылки SHALL поставить её в очередь и сообщить позицию в очереди.
6. WHEN Администратор отменяет активную рассылку, THE Модуль_Рассылки SHALL прекратить отправку новых сообщений в течение 5 секунд.

### Требование 9: Статистика и аналитика

**User Story:** Как Администратор, я хочу видеть метрики по пользователям и активности, чтобы оценивать рост и вовлечённость.

#### Acceptance Criteria

1. WHEN Администратор вызывает команду статистики, THE Модуль_Статистики SHALL вернуть: общее количество пользователей, количество активных за 24 часа / 7 дней / 30 дней, количество забаненных, количество привязанных кошельков.
2. WHEN Администратор запрашивает статистику по регистрациям за период, THE Модуль_Статистики SHALL вернуть распределение новых пользователей по дням запрошенного периода.
3. THE Модуль_Статистики SHALL считать пользователя активным за период, если его поле `last_seen_at` попадает в этот период.

### Требование 10: Мультиязычность

**User Story:** Как пользователь, я хочу видеть интерфейс бота на своём языке, чтобы им было удобнее пользоваться.

#### Acceptance Criteria

1. THE Модуль_Локализации SHALL поддерживать минимум два языка: русский (`ru`) и английский (`en`), расширяемые через файлы переводов.
2. WHEN пользователь впервые взаимодействует с ботом, THE Модуль_Локализации SHALL выбрать язык по `language_code` из Telegram, если он входит в список поддерживаемых, иначе язык по умолчанию из Конфигурации.
3. WHEN пользователь меняет язык в профиле, THE Модуль_Локализации SHALL использовать выбранный язык для всех последующих ответов бота этому пользователю.
4. IF для текущего языка отсутствует ключ перевода, THEN THE Модуль_Локализации SHALL вернуть значение ключа на языке по умолчанию.

### Требование 11: Антиспам и капча

**User Story:** Как владелец бота, я хочу защититься от ботов и спама, чтобы сохранить качество аудитории.

#### Acceptance Criteria

1. WHERE Модуль_Антиспам включён в Конфигурации, THE Telegram_Бот SHALL требовать от нового пользователя прохождения капчи при первом запуске до выполнения любых других команд.
2. IF пользователь не проходит капчу в течение 60 секунд, THEN THE Модуль_Антиспам SHALL установить `status=pending_captcha` и не пропускать его к остальным функциям до повторного прохождения.
3. WHILE пользователь отправляет более 5 сообщений за 3 секунды, THE Модуль_Антиспам SHALL временно игнорировать его сообщения в течение 30 секунд и записать событие в Журнал_Действий.
4. THE Модуль_Антиспам SHALL хранить счётчики частоты сообщений в Кэше.

### Требование 12: Проверка подписки на каналы

**User Story:** Как владелец бота, я хочу требовать подписку на заданные каналы, чтобы использовать бота как канал трафика.

#### Acceptance Criteria

1. WHERE в Конфигурации задан список обязательных каналов, THE Модуль_Подписок SHALL проверять подписку пользователя при каждом обращении к основным командам.
2. IF пользователь не подписан хотя бы на один обязательный канал, THEN THE Модуль_Подписок SHALL вернуть список каналов с кнопками и кнопкой «Проверить подписку», блокируя остальные команды.
3. WHEN пользователь нажимает «Проверить подписку» и подписан на все каналы, THE Модуль_Подписок SHALL разрешить выполнение исходной команды.
4. IF бот не имеет прав на проверку подписки в канале, THEN THE Модуль_Подписок SHALL записать ошибку в Журнал_Действий и пропустить проверку по этому каналу.

### Требование 13: Платежи (Telegram Stars и TON)

**User Story:** Как владелец бота, я хочу принимать оплату через Telegram Stars и/или TON, чтобы монетизировать функционал.

#### Acceptance Criteria

1. WHERE Модуль_Платежей включён и настроен на Telegram Stars, THE Модуль_Платежей SHALL создавать инвойс через Telegram Bot API и обрабатывать событие `successful_payment`.
2. WHERE Модуль_Платежей включён и настроен на TON, THE Модуль_Платежей SHALL создавать транзакцию оплаты через TON Connect на адрес, заданный в Конфигурации, и проверять её подтверждение через TON API.
3. WHEN оплата успешно подтверждена, THE Модуль_Платежей SHALL сохранить запись о платеже (`user_id`, `amount`, `currency`, `tx_hash_or_charge_id`, `paid_at`) и вызвать зарегистрированный обработчик успешной оплаты.
4. IF оплата не подтверждается в течение 15 минут после создания, THEN THE Модуль_Платежей SHALL пометить платёж как `expired` и сообщить пользователю об истечении срока оплаты.
5. THE Модуль_Платежей SHALL обеспечивать идемпотентность: повторная обработка одного и того же `tx_hash_or_charge_id` не должна создавать дубликаты записей.

### Требование 14: Управление диалоговыми состояниями (FSM)

**User Story:** Как разработчик на базе шаблона, я хочу описывать пошаговые диалоги с пользователем, чтобы строить формы и мастера.

#### Acceptance Criteria

1. THE Менеджер_Состояний SHALL хранить текущее состояние пользователя и связанные данные в Кэше с ключом по `telegram_id`.
2. WHEN пользователь отменяет диалог командой `/cancel`, THE Менеджер_Состояний SHALL очистить его состояние и данные в Кэше.
3. WHERE пользователь не присылает следующий шаг диалога в течение таймаута, заданного в Конфигурации (по умолчанию 30 минут), THE Менеджер_Состояний SHALL удалить состояние пользователя.
4. IF Кэш недоступен, THEN THE Telegram_Бот SHALL вернуть пользователю сообщение о временной недоступности диалогов и записать ошибку в Журнал_Действий.

### Требование 15: Модульная архитектура и конфигурация

**User Story:** Как разработчик нового бота на базе шаблона, я хочу включать и отключать модули через конфигурацию, чтобы не тащить ненужный функционал.

#### Acceptance Criteria

1. THE Конфигурация SHALL задавать флаги включения для следующих модулей: TON_Коннектор, Реферальная_Система, Модуль_Рассылки, Модуль_Статистики, Модуль_Локализации, Модуль_Антиспам, Модуль_Подписок, Модуль_Платежей.
2. WHEN флаг модуля в Конфигурации имеет значение «выключено», THE Telegram_Бот SHALL не регистрировать команды и обработчики этого модуля при старте.
3. IF обязательный секрет (токен бота, DSN Базы_Данных) отсутствует в Конфигурации на старте, THEN THE Telegram_Бот SHALL завершить запуск с ненулевым кодом возврата и сообщением, указывающим отсутствующий ключ.
4. THE Конфигурация SHALL читать значения из переменных окружения и/или файла `.env`, не хранить секреты в коде репозитория.

### Требование 16: Поэтапное подключение интеграций

**User Story:** Как разработчик, я хочу чтобы интеграции подключались поэтапно, чтобы можно было запустить MVP и постепенно расширяться.

#### Acceptance Criteria

1. THE Telegram_Бот SHALL поддерживать запуск этапа 1 (MVP) с минимальным набором интеграций: Telegram Bot API, База_Данных, Кэш, без остальных модулей.
2. THE Telegram_Бот SHALL поддерживать поэтапное подключение интеграций в следующем порядке без изменения ядра: этап 1 — Telegram Bot API + База_Данных + Кэш; этап 2 — Модуль_Авторизации и Админ_Панель; этап 3 — TON_Коннектор; этап 4 — Модуль_Локализации, Модуль_Антиспам, Модуль_Подписок; этап 5 — Реферальная_Система, Модуль_Рассылки, Модуль_Статистики; этап 6 — Модуль_Платежей.
3. THE Telegram_Бот SHALL запускаться и проходить health-check при включённых только интеграциях этапа 1.
4. WHEN добавляется новая интеграция следующего этапа, THE Конфигурация SHALL требовать только новые параметры этой интеграции, не ломая работу уже включённых модулей.

### Требование 17: Наблюдаемость и устойчивость

**User Story:** Как владелец бота, я хочу, чтобы бот переживал кратковременные сбои внешних сервисов, чтобы не падал от одной ошибки Telegram API или Базы_Данных.

#### Acceptance Criteria

1. IF запрос к Telegram Bot API завершается сетевой ошибкой (таймаут соединения, разрыв соединения или HTTP-ответ с кодом 5xx), THEN THE Telegram_Бот SHALL повторить запрос до 3 раз с задержками 1, 2 и 4 секунды между последовательными попытками.
2. IF после 3 повторных попыток запрос к Telegram Bot API завершается ошибкой, THEN THE Telegram_Бот SHALL записать в Журнал_Действий запись об ошибке с указанием идентификатора обновления и типа ошибки, и THE Telegram_Бот SHALL продолжить обработку последующих обновлений без остановки процесса.
3. WHEN внешний клиент отправляет HTTP-запрос на эндпоинт health-check, THE Telegram_Бот SHALL в течение 2 секунд вернуть ответ со статусом «доступно» или «недоступно» отдельно для Базы_Данных и отдельно для Кэша.
4. IF проверка подключения к Базе_Данных или Кэшу в рамках health-check не возвращает ответ в течение 1 секунды, THEN THE Telegram_Бот SHALL пометить соответствующий компонент как «недоступно» в ответе health-check.
5. WHEN Telegram_Бот запускается, THE Telegram_Бот SHALL записать в Журнал_Действий версии бот-фреймворка, драйвера Базы_Данных и драйвера Кэша до начала приёма обновлений от Telegram Bot API.

## Приложение А. Этапный план подключения интеграций

Этот раздел описывает какие интеграции нужны на каждом этапе. Конкретный выбор библиотек/сервисов (aiogram vs python-telegram-bot, PostgreSQL vs SQLite, Redis vs Valkey и т.п.) принимается в документе проектирования.

### Этап 1. MVP — ядро бота
Цель: рабочий бот с регистрацией, хранением пользователей, FSM.
Интеграции:
- Telegram Bot API (long polling или webhook).
- Реляционная База_Данных (PostgreSQL или SQLite для разработки).
- Кэш (Redis или in-memory для разработки) — для FSM и временных данных.
- Конфигурация через переменные окружения / `.env`.
Покрывает требования: 1, 2, 14, 15, 16, 17.

### Этап 2. Админ-панель
Цель: безопасная модерация пользователей.
Интеграции:
- Журнал_Действий — отдельная таблица в Базе_Данных.
- Список начальных Суперадминистраторов в переменных окружения.
Покрывает требования: 4, 5, 6.

### Этап 3. TON Connect
Цель: привязка TON-кошелька пользователя.
Интеграции:
- TON Connect 2 (манифест `tonconnect-manifest.json`, публичный URL).
- TON API/индексатор для проверки подтверждений транзакций (когда включается Модуль_Платежей через TON).
- HTTPS-эндпоинт для обратных вызовов/манифеста.
Покрывает требования: 3.

### Этап 4. UX и защита
Цель: комфортный и защищённый интерфейс.
Интеграции:
- Файлы переводов (i18n) в репозитории.
- Хранение счётчиков антиспама в Кэше.
- Проверка подписок через Telegram Bot API (`getChatMember`).
Покрывает требования: 10, 11, 12.

### Этап 5. Рост и аналитика
Цель: инструменты роста и отчётности.
Интеграции:
- Очередь задач для рассылки (в Кэше или отдельный брокер, если объёмы растут).
- Агрегирующие запросы в Базе_Данных (возможен отдельный материализованный слой при больших объёмах).
Покрывает требования: 7, 8, 9.

### Этап 6. Монетизация
Цель: приём платежей.
Интеграции:
- Telegram Stars — встроенный механизм Telegram Bot API (`sendInvoice`, событие `successful_payment`).
- TON-платежи — TON Connect + TON API/индексатор для подтверждения транзакции.
- Опционально: webhook получения событий от платёжного провайдера.
Покрывает требования: 13.

### Этап 7. Эксплуатация
Цель: стабильный прод.
Интеграции:
- Хостинг с поддержкой Python/Node (VPS, Docker, Fly.io, Railway, собственный сервер).
- CI/CD (например, GitHub Actions) — сборка, тесты, деплой.
- Мониторинг логов (stdout → агрегатор на выбор).
- Health-check эндпоинт, упомянутый в Требовании 17.
